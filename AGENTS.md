# AGENTS.md

## 项目说明
本仓库用于开发 `OrderManagerSystem`，这是一个基于桌面端的订单管理系统，重点功能包括：

- 订单录入
- 配置模板选择与自定义组件录入
- 订单级与组件级发货跟踪
- 订单查询与订单汇总输出

本项目是**轻量级业务系统**，不是完整 ERP。

---

## 技术栈约束
除非用户明确要求，否则始终遵守以下技术路线：

- C++
- Qt 6
- Qt Widgets
- CMake
- SQLite

约束规则：

- 不要改用 QML
- 不要改用 Web 技术栈
- 不要引入大型第三方框架
- 优先使用 Qt 自带能力

---

## 读取顺序
在进行中等及以上复杂度任务前，先阅读以下文件：

1. `requirements.md`
2. `AGENTS.md`
3. `plan.md`

说明：

- `requirements.md`：说明系统要做什么
- `AGENTS.md`：说明本仓库的实现规则
- `plan.md`：说明当前阶段先做什么

## 实现规则
所有代码修改默认遵守以下规则：

- 优先做**最小可运行版本**
- 改动尽量**小、清晰、可审查**
- 不要一次性大范围重构
- 不要修改无关代码
- 不要把 UI、数据库、业务逻辑混在一个文件里
- 优先保证代码可读、可维护
- 未经明确要求，不提前实现后续里程碑内容

实现顺序建议：

1. 先让功能能跑起来
2. 再整理代码结构
3. 最后再考虑扩展性

---

## Git 与改动控制
本项目使用 Git 管理。

规则：

- 保持 diff 尽量小
- 不修改无关文件
- 不提交、不改写 Git 历史，除非用户明确要求
- 尽量让每次改动都容易 review

一般不要主动修改这些内容：

- `build/`
- `.qtcreator/`
- 其他本地生成文件
- IDE 本地配置文件

---

## 架构约束
即使当前文件还在根目录，也应按分层思路实现：

- UI 层：窗口、对话框、控件交互
- 数据库层：连接、建表、查询、持久化
- 业务逻辑层：订单处理、配置展开、发货更新
- 数据模型层：订单、组件、模板、发货记录等结构

避免以下做法：

- 把所有逻辑都写进 `MainWindow`
- 在 UI 代码中到处直接写 SQL
- 让控件逻辑直接承担复杂业务判断

---

## 数据库规则
数据库是本项目核心部分，默认使用 SQLite。

必须遵守：

- 建表逻辑要明确、集中
- 关系字段要清晰
- 保留历史记录，不要只保留聚合结果
- 模板变更不能反向影响历史订单
- 订单最终实际组件必须显式保存

必须保持的业务约束：

- 一个客户可以有多条订单项
- 订单项默认独立跟踪
- 发货记录应保留历史
- 配置模板只是录单辅助，不能替代实际订单组件数据

---

## UI 规则
使用 Qt Widgets 实现界面。

界面实现原则：

- 实用优先
- 录入效率优先
- 表格与表单应清晰易操作
- 优先使用标准 Qt 控件
- 不追求复杂炫酷效果

---

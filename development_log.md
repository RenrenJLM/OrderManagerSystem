# OrderManagerSystem 开发日志

## 2026-04-19：订单录入默认启动与订单更新链路校正

### 本轮目标
- 将主窗口启动后的默认页稳定落到 `订单录入`。
- 全面复核当前新业务模型下“录单 -> 发货 -> 库存 -> 查询”的更新链路。
- 修正会导致订单价格、数量或库存匹配失真的关键问题，并补最小回归测试。

### 本轮实现
- 启动页行为调整：
  - 主窗口完成动态页签插入后，显式切换到 `订单录入` 页。
  - 不再依赖 `.ui` 中的静态 `currentIndex` 推断启动页。
- 订单保存口径收紧：
  - `saveStructuredOrder(...)` 改为按最终订单组件快照重算 `config_price`。
  - 灯罩单价优先取最终灯罩组件快照，避免订单头价格与订单组件快照脱节。
  - 订单头价格不再直接信任界面传入值。
- 库存自然键修正：
  - 库存匹配、库存导入覆盖、整单可发货判定、发货扣减统一纳入 `unit_name`。
  - 修正同名物料但不同单位时可能发生的错配、串扣问题。
- 发货与页面联动收口：
  - 录单保存、整单发货、组件发货、库存保存、库存导入后统一刷新：
    - 库存页
    - 可发货提示
    - 发货页订单源
    - 订单查询结果
  - 发货写回时同步更新 `order_components.updated_at` 与库存更新时间。
- 录单组件编辑稳定性修正：
  - 组件行从库存候选切回空项或手输项时，清理候选残留的规格、材质、颜色、单位和单价，避免脏数据被带入订单快照。
- 自动化回归补充：
  - 新增 `OrderManagerSystemTests`。
  - 覆盖启动默认页、订单价格按组件重算、库存按单位匹配并扣减等核心场景。

### 当前结果
- 当前正式主链路已实现：
  - 启动默认进入订单录入
  - 订单保存时同步固化最终组件与订单价格
  - 库存按物料 + 单位参与匹配
  - 录单、发货、库存维护后的订单与库存相关页面联动刷新
- 当前业务口径下，历史订单快照不会被后续主数据或库存修改反向改写。

## 2026-04-19：数据库文件管理能力

### 本轮目标
- 为当前正式版系统增加单库模式下的：
  - 新建数据库
  - 打开已有数据库
  - 关闭当前数据库
- 严格只面向当前正式版新业务模型，不兼容旧模型，不做迁移。

### 本轮实现
- 数据库层新增显式生命周期接口：
  - `createDatabaseFile(...)`
  - `openDatabaseFile(...)`
  - `closeDatabase()`
  - `isDatabaseOpen()`
  - `currentDatabasePath()`
  - `validateDatabaseFile(...)`
- 新建数据库时：
  - 只创建当前正式版所需的新模型表结构
  - 不再自动写入默认主数据、BOM 或库存
- 打开数据库时：
  - 先校验必需表与关键字段
  - 缺表或缺字段时给出正式中文错误并拒绝切换
  - 不做隐式补表、补列或迁移
- 主窗口新增文件菜单入口：
  - `新建数据库...`
  - `打开数据库...`
  - `关闭当前数据库`
- 主窗口标题区新增当前数据库状态显示：
  - 已打开时显示绝对路径
  - 未打开时显示“当前数据库：未打开”
- 启动行为调整为：
  - 记住上次成功打开的数据库路径
  - 启动时自动尝试恢复
  - 恢复失败则进入未打开数据库安全状态
- 关闭数据库后统一执行：
  - 清空页面缓存和表格数据
  - 禁用录单、发货、导入、导出、打印和主数据保存相关操作
  - 状态提示切换为正式未打开文案

### 验证口径
- 本轮仍以 Windows `Qt 6.7.3 + MSVC 2019` 手动验证为准。
- 本地已补充代码级静态检查，并保持当前订单、库存、发货核心口径不变。

## 2026-04-18：Milestone 6 导入容错修正与产品类型文案统一

### 本轮问题
- 在导入 `base_configuration_bom_import.csv` 时，部分 BOM 行虽然数量和金额实际已填写，但由于中间描述列缺失、CSV 空列数量不稳定，导致尾部 `quantity` / `unit_amount` / `sort_order` 被错位解析。
- 当前界面与错误提示中仍残留“产品大类型”“大类型”等旧文案，不符合当前业务表达。

### 本轮修正
- 调整 `DataImporter` 中 BOM 与库存导入的行解析策略：
  - 不再完全依赖固定列索引直接取值
  - 改为“前缀固定 + 尾部关键列固定 + 中间描述列容错补空”的方式解析
  - 对 BOM 行，优先保证：
    - `unit_name`
    - `quantity`
    - `unit_amount`
    - `sort_order`
  - 对库存行，优先保证：
    - `unit_name`
    - `unit_price`
    - `current_quantity`
    - `note`
- 当前导入口径调整为：
  - 规格、材质、颜色、描述类字段允许为空
  - 只要价格计算和库存计算所依赖的关键列合法，就允许导入
  - 若关键尾部列本身缺失或数值非法，仍然报错
- 统一当前运行代码中的用户可见文案：
  - `产品大类型` 改为 `产品类型`
  - `大类型` 改为 `类型`
  - 覆盖范围包括：
    - 产品资料维护页
    - 库存页
    - 导入提示
    - 数据库层错误提示

### 影响与结果
- 类似 `飞月,C,小喇叭,,,铁,白色,件,1,0.40,8` 的 BOM 行现在可被正确解析，不再把 `quantity` 误读为 `件`。
- 导入功能对真实业务表格中的“描述列不稳定”情况更宽容，但仍保持对数量、金额、单价、库存等关键字段的严格校验。
- 当前界面、导入结果提示和错误信息已统一使用“产品类型”口径。

## 2026-04-18：Milestone 6 最小导入能力落地

### 本轮目标
- 在不改变 Milestone 5 新业务模型和现有稳定主流程的前提下，补齐主数据与库存初始化的最小批量导入能力。
- 严格只围绕当前真实新模型实现：
  - `product_categories`
  - `product_skus`
  - `base_configurations`
  - `base_configuration_components`
  - `inventory_items`
- 不兼容旧模板模型，不引入通用导入框架，不实现 Excel 解析。

### 本轮实现
- 新增独立导入模块 `DataImporter`：
  - 固定使用 UTF-8 CSV
  - 按固定列头解析
  - 提供按类型分开的导入函数
  - 汇总成功 / 跳过 / 失败数量与失败原因摘要
- 在 `DatabaseManager` 中补充导入所需的新模型辅助接口：
  - 按产品大类型名称查找 `product_category_id`
  - 按产品大类型 + 配置代码查找基础配置
  - 产品大类型按名称覆盖写入
  - 具体型号按 `(product_category_id, sku_name)` 覆盖写入
  - 基础配置按 `(product_category_id, config_code)` 覆盖写入

- 对库存导入增加重复自然键防护：
  - 若现有库存中同一自然键命中多条记录，则拒绝自动覆盖并返回错误，避免误更新
- 基础配置 BOM 导入按“单个基础配置整组替换”处理：
  - 先按 `产品大类型 + 配置代码` 分组
  - 单组数据校验通过后调用现有 `replaceBaseConfigurationComponents(...)`
  - 保持 BOM 替换边界清晰，不产生半套 BOM 落库
- 在 UI 中增加最小导入入口：
  - `产品资料维护` 四个维护页各自增加 `导入 CSV` 按钮
  - `库存管理 -> 库存录入` 增加 `导入初始库存 CSV` 按钮
  - 导入完成后展示结果摘要，并自动刷新主数据、库存、查询和发货提示相关页面

### 当前导入口径
- 仅支持 CSV，不支持 Excel。
- 仅支持固定列头，不做动态列识别。
- 所有导入字段严格映射当前新模型表字段，不新增临时字段。
- 当前覆盖策略：
  - 产品大类型：按名称覆盖
  - 具体型号：按所属大类型 + 型号名称覆盖
  - 基础配置：按所属大类型 + 配置代码覆盖
  - BOM：按所属大类型 + 配置代码整组覆盖
  - 库存：按自然键覆盖

## 2026-04-18：稳定性审查与对象生命周期修正

### 审查结论
- 本轮重点检查了近期新增的动态 UI 结构，尤其是库存管理页和发货登记页的页签重组代码。
- 已确认并修复 2 处明确的对象生命周期问题：
  - 发货页使用 `shipmentLayout->takeAt()` 重组布局时，取出的 `QLayoutItem` 未被安全移交，存在泄漏风险。
  - 订单录入页移除 `priceRowSpacer` 后未释放，存在稳定的 `QSpacerItem` 泄漏。

### 本轮修正
- 发货页页签重组改为标准的 Qt 迁移方式：
  - 对按钮区和组件发货表单使用 `removeItem(...)`
  - 对组件表使用 `removeWidget(...)`
  - 再挂入新的页签布局
- 订单录入页在移除 `priceRowSpacer` 后立即 `delete`，并清空指针。
- 订单查询页改为“订单查询 / 订单详情”双页签结构：
  - 列表页仅展示订单列表
  - 详情页展示最终订单组件和发货记录
  - 点击订单列表行会自动跳转到详情页
- 查询页详情区当前调整为上下两个表格：
  - 上方“最终订单组件”
  - 下方“发货记录”
- 修正结构化订单查询表在启用排序时逐行插入导致的偶发列内容错位问题，查询结果现先填充再排序。
- 订单录入页“产品大类型”文案调整为“产品类型”。
- 结合前几轮改动，库存管理当前结构为：
  - `库存录入`
  - `当前库存列表`
  - `未发货订单物料需求`
- “未发货订单物料需求”页当前支持：
  - 上框显示未满足发货条件的订单
  - 下框支持查看全部未满足订单总缺料，或查看单个订单缺料

### 当前结论
- 当前代码中最明确的 UI 生命周期泄漏点已修正。
- 已完成实际运行验证，当前版本可作为后续开发基线。

## Milestone 3：发货功能

### 阶段目标
- 实现订单级和组件级的最小发货登记闭环。
- 支持同一订单、多次追加发货，并保留每次发货历史。
- 正确维护订单和订单组件的已发/未发数量，不覆盖原始订单组件快照。

### 本阶段完成内容
- 复用现有 `shipment_records` 单表实现发货历史保存，未新增 `component_shipment_records`。
- 在数据库层新增最小发货相关接口：
  - 读取订单级发货列表
  - 读取指定订单的组件发货状态
  - 保存订单级发货记录
  - 保存组件级发货记录
- 在主窗口新增最小发货登记区域，包含：
  - 发货订单选择
  - 订单状态显示
  - 订单级发货输入与保存
  - 组件状态表
  - 选中组件后的组件级发货输入与保存
- 订单级发货采用“追加记录 + 同步组件”模式：
  - 每次按套数登记发货时，向 `shipment_records` 新增一条订单级发货记录
  - 同时按订单组件快照同步增加各组件已发数量并扣减未发数量
- 组件级发货采用“追加记录 + 重算订单状态”模式：
  - 每次按组件数量登记发货时，向 `shipment_records` 新增一条组件级发货记录
  - 更新目标组件已发/未发数量
  - 再根据所有组件当前状态重算订单已发/未发套数，避免订单级和组件级口径漂移

### 已实现的业务规则
- 发货记录始终采用追加写入，不覆盖历史记录。
- 订单级发货记录使用：
  - `order_item_id` 指向订单
  - `order_item_component_id = NULL`
- 组件级发货记录使用：
  - `order_item_id` 指向订单
  - `order_item_component_id` 指向具体订单组件
- 订单级发货会校验：
  - 发货套数必须大于 0
  - 发货套数不能超过当前组件状态允许的可发范围
- 组件级发货会校验：
  - 发货数量必须大于 0
  - 发货数量不能超过该组件未发数量
- 原始订单组件快照字段不会因发货被改写：
  - `component_name`
  - `quantity_per_set`
  - `total_required_quantity`
  - `source_type`

### 验证结果
- 已在代码层完成 Milestone 3 所需 UI、数据库接口与事务保存逻辑。
- 本项目的编译、运行和手工功能验证统一在 Windows `Qt 6.7.3 + MSVC 2019` 环境中完成。

### 遇到的问题 1：单表发货记录如何同时支撑订单级和组件级
- 现有表 `shipment_records` 同时具备：
  - `order_item_id`
  - `order_item_component_id`
- 这意味着它已经可以区分：
  - 面向订单的发货记录
  - 面向组件的发货记录
- 因此如果再新增 `component_shipment_records`，会增加不必要的结构复杂度。

### 解决方案 1
- 保持 `shipment_records` 单表设计不变。
- 通过 `order_item_component_id` 是否为空区分订单级 / 组件级发货记录。
- 这样既满足“追加记录、不覆盖历史”，又能保持本轮改动最小。

### 遇到的问题 2：组件级发货后，订单级已发/未发套数可能与组件状态不一致
- 如果组件级发货只更新 `order_item_components`，不处理 `order_items`，则：
  - 订单级未发套数会停留在旧值
  - 后续订单级发货可能把部分组件推进到超发边界
- 这会导致订单级和组件级的发货状态口径漂移。

### 解决方案 2
- 每次组件级发货后，根据当前订单全部组件的已发/未发数量重算：
  - `shipped_sets`
  - `unshipped_sets`
- 重算方式采用“按每套数量折算后的最小完整套数”，保证订单级状态始终与组件级状态一致。
- 订单级发货后也同样基于组件状态回写订单套数，确保两套数据口径统一。

### 本阶段结论
- Milestone 3 已按当前范围实现最小发货登记闭环。
- 当前实现只覆盖：
  - 订单级发货登记
  - 组件级发货登记
  - 发货历史追加落库
  - 已发/未发数量同步更新
- 未提前实现查询、筛选、汇总、导出、打印、权限、网络同步或 Milestone 4 内容。

### 补充修复记录：发货口径与退出稳定性
- 根据后续联调反馈，补充修复了两个问题：
  - 组件级发货后，订单级 `已发套数/未发套数` 被普通组件发货错误污染，导致后续“按套数发货”判断失真。
  - 程序退出时，发货组件表的选择变化回调可能在析构阶段继续访问控件，触发 Debug 环境下的 heap corruption。
- 修复方式：
  - 为每张订单自动维护一条隐含“机体”组件记录，使用 `source_type = system_body`。
  - `order_items.shipped_sets / unshipped_sets` 只表示机体本身发货计数，不再由普通组件发货反推。
  - “可按套再发数量”改为由全部组件当前剩余量推导，发货界面按该值控制按套发货。
  - 启动时自动修复历史订单：
    - 为旧订单补齐机体组件
    - 按订单级发货历史重算 `order_items`
    - 同步机体组件的已发/未发数量
- 在主窗口析构开始时断开发货组件表相关信号，并为发货选择更新逻辑补充关闭期与空指针保护。

### 补充修复记录 2：订单完成状态显示与机体历史恢复
- 继续联调后发现两个问题：
  - 当机体已发完但仍有组件未发完时，界面无法明确显示订单是否真正完成。
  - 启动时的历史修复逻辑只统计了订单级发货记录，没有把“机体组件发货记录”一起纳入机体恢复，导致重启后 `order_items` 与机体组件数量被回退到较小值。
- 修复方式：
  - 在订单发货摘要中补充“订单是否完成”的派生状态。
  - 发货区状态文案改为同时显示：
    - 机体已发/未发
    - 可按套再发
    - 订单状态（已完成 / 机体已发完但组件未完 / 组件不足以继续整套发货 / 未完成）

### 遇到的问题
- 当前订单主表中的 `shipped_sets / unshipped_sets` 语义已在 Milestone 3 中收敛为机体维度，而订单“完成状态”需要基于全部组件是否发完来判断。

### 解决方案
- 订单列表状态改为在查询 SQL 中通过 `order_item_components` 派生：
  - 当该订单全部组件 `unshipped_quantity = 0` 时判定为 `已完成`
  - 否则判定为 `未完成`
- 这样不会影响 Milestone 3 的机体发货口径，同时能保证查询页状态和实际订单完成情况一致。

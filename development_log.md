## Milestone 1：数据库初始化与建表

### 阶段目标
- 程序启动时自动创建 SQLite 数据库文件。
- 初始化 6 张核心表：
  - `product_models`
  - `option_templates`
  - `option_template_components`
  - `order_items`
  - `order_item_components`
  - `shipment_records`
- 将建表逻辑集中在数据库层，不分散到 UI 代码中。

### 本阶段完成内容
- 在 `DatabaseManager` 中完成数据库连接、外键启用、建表语句执行。
- 程序启动时先完成数据库初始化，再显示主窗口。
- 核心业务表已能重复启动自动检查并保持可用，不会因重复建表报错。

### 验证结果
- 在 Windows `Qt 6.7.3 + MSVC 2019` 环境下完成编译与运行验证。
- 运行程序后成功生成 `OrderManagerSystem.db`。
- 数据库中确认存在 6 张核心表。

### 遇到的问题
- 早期曾在 WSL 命令行中尝试执行 `cmake -S . -B build`。
- 但本项目实际开发方式是：
  - 使用 WSL 作为 Codex 辅助分析与代码修改环境
  - 使用 Windows 下的 Qt Creator / Qt 工具链进行实际编译、运行和功能验证
- 因此 WSL 本身并不是本项目的目标构建验证环境。
- 同时，WSL 中可见的 Qt 版本为 `6.2.4`，也不满足项目 `CMakeLists.txt` 中 `Qt 6.5` 的要求。

### 解决方案
- 明确将 WSL 仅作为 Codex 辅助环境，不将其作为 Qt 项目的实际构建验证环境。
- 实际编译、运行与功能验证统一放在 Windows 下已经安装并可工作的 `Qt 6.7.3 + MSVC 2019` 环境中完成。
- 后续涉及 Qt/CMake 构建验证，统一以 Windows Qt Creator 构建结果为准，而不是以 WSL 命令行结果为准。

### 本阶段结论
- Milestone 1 已完成。
- 数据库初始化、建表和启动入口已打通，为后续录单闭环提供了稳定的数据基础。

## Milestone 2：订单录入最小闭环

### 阶段目标
- 实现“录入订单 -> 选择模板或自定义配置 -> 展开订单组件 -> 保存订单”的最小闭环。
- 模板仅作为录单辅助，最终订单组件必须以快照方式写入 `order_item_components`。

### 本阶段完成内容
- 在主窗口实现最小可用录单界面，包含：
  - 订单日期
  - 客户名称
  - 产品型号选择
  - 配置模板选择
  - 订单套数
  - 单价
  - 模板配置 / 自定义配置切换
  - 订单组件表格
- 在数据库层新增最小录单相关接口：
  - 读取产品型号
  - 按产品读取模板
  - 读取模板组件
  - 保存订单主记录
  - 保存订单组件快照
- 增加最小演示数据写入逻辑，保证产品型号与模板可直接用于界面联调。
- 实现订单保存时的事务处理，先写 `order_items`，再写 `order_item_components`。

### 已实现的业务规则
- 模板组件会展开为订单组件快照。
- 自定义组件只写当前订单，不回写模板表。
- 订单组件总需求数量按“每套数量 × 订单套数”计算。
- 订单初始发货状态为未发：
  - `shipped_sets = 0`
  - `unshipped_sets = quantity_sets`
- 组件初始发货状态为未发：
  - `shipped_quantity = 0`
  - `unshipped_quantity = total_required_quantity`
- 同名组件的最终合并保证放在数据库保存前处理。

### 验证结果
- 在 Windows `Qt 6.7.3 + MSVC 2019` 下可以成功编译运行。
- 当前数据库中已确认存在产品、模板、模板组件基础数据。
- 已成功保存多条模板配置订单和自定义配置订单。
- 已检查 `order_items` 与 `order_item_components`：
  - 模板订单组件快照与模板明细一致
  - 自定义订单组件以 `manual` 来源写入
  - 总需求数量计算正确
  - 未发数量初始值正确

- 同时，WSL 中的 Qt 版本也不满足项目要求，不适合作为参考构建结果。

### 解决方案 1

- WSL 只承担辅助作用，不再把 WSL 中的 Qt/CMake 结果作为功能是否完成的判断依据。

### 遇到的问题 2：程序关闭时出现 MSVC Debug Error
- 现象：
  - 点击右上角关闭按钮时，MSVC Debug 运行库报错：
  - `Run-Time Check Failure #2 - Stack around the variable 'w' was corrupted`
- 初始表现为：
  - 即使不做任何操作，启动后直接关闭也会报错。

### 问题分析
- 报错点出现在 `main.cpp` 中的主窗口栈对象 `w`，但实际问题并不一定在 `main.cpp` 本身。
- 重点排查后发现风险主要集中在：
  - `MainWindow` 的初始化/销毁生命周期
  - `QTableWidget` 相关信号回调
  - UI 更新时可能发生的重入修改

### 解决方案 2
- 将主窗口从栈对象改为堆对象持有，减少运行时检查只在 `w` 上暴露问题的情况。
- 为 `MainWindow` 增加初始化中和关闭中的状态保护：
  - `m_isInitializing`
  - `m_isShuttingDown`
- 收紧组件表更新路径：
  - 不再在危险回调中重建整张表
  - 表格批量更新时统一使用 `QSignalBlocker`
  - 删除组件时不再触发表格重建
- 在关键位置加入最小 `qDebug()` 输出，辅助确认启动和关闭链路。

### 修复后验证结果
- 关闭窗口时不再出现 Debug Error。
- 应用输出中可正常看到：
  - `MainWindow initialized`
  - `Showing main window`
  - `MainWindow shutting down`
- 关闭进程返回成功。
- 添加组件等基本操作也未再触发退出时报错。

### 本阶段结论
- Milestone 2 已基本完成并通过当前范围内验证。
- 订单录入、模板展开、自定义组件录入、订单保存与组件快照保存链路已打通。
- 当前实现仍保持在最小闭环范围内，未提前实现发货、查询或汇总功能。

## Milestone 3：发货功能

### 阶段目标
- 实现订单级和组件级的最小发货登记闭环。
- 支持同一订单、多次追加发货，并保留每次发货历史。
- 正确维护订单和订单组件的已发/未发数量，不覆盖原始订单组件快照。

### 本阶段完成内容
- 复用现有 `shipment_records` 单表实现发货历史保存，未新增 `component_shipment_records`。
- 在数据库层新增最小发货相关接口：
  - 读取订单级发货列表
  - 读取指定订单的组件发货状态
  - 保存订单级发货记录
  - 保存组件级发货记录
- 在主窗口新增最小发货登记区域，包含：
  - 发货订单选择
  - 订单状态显示
  - 订单级发货输入与保存
  - 组件状态表
  - 选中组件后的组件级发货输入与保存
- 订单级发货采用“追加记录 + 同步组件”模式：
  - 每次按套数登记发货时，向 `shipment_records` 新增一条订单级发货记录
  - 同时按订单组件快照同步增加各组件已发数量并扣减未发数量
- 组件级发货采用“追加记录 + 重算订单状态”模式：
  - 每次按组件数量登记发货时，向 `shipment_records` 新增一条组件级发货记录
  - 更新目标组件已发/未发数量
  - 再根据所有组件当前状态重算订单已发/未发套数，避免订单级和组件级口径漂移

### 已实现的业务规则
- 发货记录始终采用追加写入，不覆盖历史记录。
- 订单级发货记录使用：
  - `order_item_id` 指向订单
  - `order_item_component_id = NULL`
- 组件级发货记录使用：
  - `order_item_id` 指向订单
  - `order_item_component_id` 指向具体订单组件
- 订单级发货会校验：
  - 发货套数必须大于 0
  - 发货套数不能超过当前组件状态允许的可发范围
- 组件级发货会校验：
  - 发货数量必须大于 0
  - 发货数量不能超过该组件未发数量
- 原始订单组件快照字段不会因发货被改写：
  - `component_name`
  - `quantity_per_set`
  - `total_required_quantity`
  - `source_type`

### 验证结果
- 已在代码层完成 Milestone 3 所需 UI、数据库接口与事务保存逻辑。
- 已尝试在当前 WSL 环境执行：
  - `cmake -S . -B build`
- 结果未能完成配置，原因不是项目逻辑错误，而是当前 WSL 环境缺少符合要求的 Qt 6 CMake 包：
  - 未找到 `Qt6Config.cmake`
  - 项目要求 `Qt 6.5+`
- 根据本项目既定约束，WSL 仅作为辅助分析环境，该失败不作为正式功能验证结论。
- 本轮仍需在 Windows `Qt 6.7.3 + MSVC 2019` 环境下完成最终编译运行与手工功能验证。

### 遇到的问题 1：单表发货记录如何同时支撑订单级和组件级
- 现有表 `shipment_records` 同时具备：
  - `order_item_id`
  - `order_item_component_id`
- 这意味着它已经可以区分：
  - 面向订单的发货记录
  - 面向组件的发货记录
- 因此如果再新增 `component_shipment_records`，会增加不必要的结构复杂度。

### 解决方案 1
- 保持 `shipment_records` 单表设计不变。
- 通过 `order_item_component_id` 是否为空区分订单级 / 组件级发货记录。
- 这样既满足“追加记录、不覆盖历史”，又能保持本轮改动最小。

### 遇到的问题 2：组件级发货后，订单级已发/未发套数可能与组件状态不一致
- 如果组件级发货只更新 `order_item_components`，不处理 `order_items`，则：
  - 订单级未发套数会停留在旧值
  - 后续订单级发货可能把部分组件推进到超发边界
- 这会导致订单级和组件级的发货状态口径漂移。

### 解决方案 2
- 每次组件级发货后，根据当前订单全部组件的已发/未发数量重算：
  - `shipped_sets`
  - `unshipped_sets`
- 重算方式采用“按每套数量折算后的最小完整套数”，保证订单级状态始终与组件级状态一致。
- 订单级发货后也同样基于组件状态回写订单套数，确保两套数据口径统一。

### 本阶段结论
- Milestone 3 已按当前范围实现最小发货登记闭环。
- 当前实现只覆盖：
  - 订单级发货登记
  - 组件级发货登记
  - 发货历史追加落库
  - 已发/未发数量同步更新
- 未提前实现查询、筛选、汇总、导出、打印、权限、网络同步或 Milestone 4 内容。

### 补充修复记录：发货口径与退出稳定性
- 根据后续联调反馈，补充修复了两个问题：
  - 组件级发货后，订单级 `已发套数/未发套数` 被普通组件发货错误污染，导致后续“按套数发货”判断失真。
  - 程序退出时，发货组件表的选择变化回调可能在析构阶段继续访问控件，触发 Debug 环境下的 heap corruption。
- 修复方式：
  - 为每张订单自动维护一条隐含“机体”组件记录，使用 `source_type = system_body`。
  - `order_items.shipped_sets / unshipped_sets` 只表示机体本身发货计数，不再由普通组件发货反推。
  - “可按套再发数量”改为由全部组件当前剩余量推导，发货界面按该值控制按套发货。
  - 启动时自动修复历史订单：
    - 为旧订单补齐机体组件
    - 按订单级发货历史重算 `order_items`
    - 同步机体组件的已发/未发数量
- 在主窗口析构开始时断开发货组件表相关信号，并为发货选择更新逻辑补充关闭期与空指针保护。

### 补充修复记录 2：订单完成状态显示与机体历史恢复
- 继续联调后发现两个问题：
  - 当机体已发完但仍有组件未发完时，界面无法明确显示订单是否真正完成。
  - 启动时的历史修复逻辑只统计了订单级发货记录，没有把“机体组件发货记录”一起纳入机体恢复，导致重启后 `order_items` 与机体组件数量被回退到较小值。
- 修复方式：
  - 在订单发货摘要中补充“订单是否完成”的派生状态。
  - 发货区状态文案改为同时显示：
    - 机体已发/未发
    - 可按套再发
    - 订单状态（已完成 / 机体已发完但组件未完 / 组件不足以继续整套发货 / 未完成）
  - 启动修复逻辑改为同时统计：
    - 订单级发货记录
    - 机体组件发货记录
  - 这样重启后 `order_items` 与机体组件都能恢复到一致的真实机体发货数量。

## 阶段 4：查询与汇总（最小版本）

### 本阶段目标
- 在不改动前 3 个里程碑核心录单与发货逻辑的前提下，补充一个最小可用查询闭环。
- 查询能力只覆盖：
  - 订单列表查看
  - 订单组件明细查看
  - 订单发货记录查看
  - 基础筛选（客户名称、产品型号、仅未完成）
- 不扩展报表、统计、分页、排序系统、导出打印、多页面或 Tab。

### 设计判断
- 现有数据库结构已经足够支撑 Milestone 4 最小范围：
  - `order_items` 提供订单列表所需主字段
  - `order_item_components` 提供订单组件明细
  - `shipment_records` 提供发货历史

### 遇到的问题
- 当前订单主表中的 `shipped_sets / unshipped_sets` 语义已在 Milestone 3 中收敛为机体维度，而订单“完成状态”需要基于全部组件是否发完来判断。
- 如果直接沿用 `order_items.unshipped_sets == 0` 判断查询状态，会把“机体已发完但组件未发完”的订单误判为已完成。

### 解决方案
- 订单列表状态改为在查询 SQL 中通过 `order_item_components` 派生：
  - 当该订单全部组件 `unshipped_quantity = 0` 时判定为 `已完成`
  - 否则判定为 `未完成`
- 这样不会影响 Milestone 3 的机体发货口径，同时能保证查询页状态和实际订单完成情况一致。

### 补充修复记录：关闭阶段异常终止
- 后续验证中，虽然未再稳定复现 CRT 对话框，但 Qt Creator 仍显示：
  - `MainWindow shutting down`
  - `terminated abnormally`
- 这说明程序在关闭阶段仍存在异常退出，不应视为正常结束。
- 本次进一步收敛析构策略：
  - 保留 `m_isShuttingDown = true`
  - 保留关键控件 `disconnect`
  - 移除析构中对多个 `QTableWidget` 的主动 `clearContents()` 与 `setRowCount(0)`
- 原因是当前录单组件表已包含 `QComboBox` 单元格控件，关闭阶段如果主窗口再主动清空表格，容易与 Qt 子控件销毁链交叉，触发二次访问或越界写风险。
- 调整后优先交由 `delete ui` 统一接管子控件销毁，减少析构期间的主动 UI 变更。

# OrderManagerSystem 分阶段实施计划

## 规划原则
- 遵循 `requirements.md` 与 `AGENTS.md`，技术栈保持 `C++ / Qt 6 / Qt Widgets / CMake / SQLite`。
- Milestone 1 至 Milestone 4 已完成，后续规划以现有可运行程序为基础，不推翻已验证的录单、发货、查询框架。
- 自 Milestone 5 起，业务模型由旧的“产品型号 + 模板配置”升级为“产品大类型 + 具体型号 + 基础配置 + 标准 BOM + 订单级附加配置 + 物料库存”。
- 当前优先级：可运行 > 完美设计 > 扩展性。
- 不提前实现复杂 ERP 能力，如权限、审批、多仓、多级报表、网络同步。

## Milestone 5A：产品配置体系 + 新下单系统

### 当前状态
- 已完成并验证通过。
- 当前系统已统一使用新业务模型：
  - `product_categories`
  - `product_skus`
  - `base_configurations`
  - `base_configuration_components`
  - `orders`
  - `order_components`
- 当前数据库不再保留此前旧数据，后续开发以现有新模型为唯一有效链路。

### 阶段目标
- 建立新的产品主数据体系。
- 将下单流程调整为：
  - 选择具体型号
  - 自动确定所属产品大类型
  - 选择该类型支持的基础配置
  - 自动带出灯罩与灯罩单价
  - 自动带出配置价格
  - 自动生成基础 BOM 清单
  - 支持订单级附加配置调整
- 形成“主数据维护 + 录单保存 + 查询回看”的新业务闭环。

### 涉及模块
- 数据库层：
  - 新产品主数据表
  - 新订单结构
  - 订单组件快照与附加配置支持
- UI 层：
  - 产品基础资料维护页
  - 新录单页
  - 查询页字段适配
- 业务逻辑层：
  - 型号关联产品大类型
  - 基础配置可选范围校验
  - 标准 BOM 展开
  - 附加配置合并与保存

### 数据结构依赖
- 必须先有：
  - 产品大类型
  - 具体型号
  - 大类型支持的基础配置及价格
  - 基础配置 BOM 模板
- 下单保存依赖：
  - 型号可反查产品大类型
  - 订单保存时生成订单组件快照
- 订单组件至少区分：
  - 基础 BOM 组件
  - 灯罩组件
  - 附加新增组件
  - 调整后的最终组件

### UI 任务拆分
- 产品资料维护：
  - 产品大类型管理
  - 具体型号管理
  - 基础配置价格维护
  - 基础配置 BOM 维护
- 订单录入：
  - 型号选择
  - 产品大类型自动显示
  - 基础配置选择
  - 灯罩信息自动带出
  - 配置价格自动带出
  - 订单组件表自动生成
  - 订单级附加配置编辑
- 查询适配：
  - 订单列表显示具体型号、产品大类型、基础配置
  - 订单明细显示最终订单组件

### 开发顺序
1. 建立新主数据表与最小样例数据。
2. 完成主数据维护最小页面。
3. 完成 BOM 展开与新订单保存逻辑。
4. 完成新录单页。
5. 适配查询页。
6. 在现有新模型基础上继续迭代后续功能。

### MVP（最小可运行路径）
- 手工维护产品大类型、具体型号、基础配置价格、基础 BOM。
- 可新增订单：
  - 选具体型号
  - 选基础配置
  - 自动生成 BOM
  - 手工做附加配置调整
  - 保存订单和订单组件快照
- 可在查询页回看订单与组件明细。
- 当前不要求：
  - 常用附加组件库
  - 批量导入
  - 历史数据迁移工具
  - 复杂统计报表

### 完成标准
- 新订单链路不再依赖旧 `option_templates` 模型。
- 每个订单可明确保存：
  - 产品大类型
  - 具体型号
  - 基础配置类型
  - 灯罩信息
  - 配置价格
  - 最终订单组件快照
- 附加配置只影响当前订单，不回写标准 BOM。
- 查询页可回看最终订单组件。

## Milestone 5B：库存管理 + 发货提示

### 当前状态
- 已完成当前版本主流程，并已通过实际运行验证。
- 当前系统已具备：
  - 按物料维度维护库存主数据
  - 未发订单物料需求统计
  - 可发货 / 不可发货提示
  - 新订单整套发货与单组件发货
  - 发货时同步扣减库存
- 近期稳定性审查已修复发货页动态重组中的布局项泄漏风险。
- 当前数据库与运行数据均以 Milestone 5 新模型为准，不再保留此前旧数据。

### 阶段目标
- 建立按“物料/组件”维度的库存管理能力。
- 支持库存录入与库存查询。
- 统计全部未发货订单的物料需求。
- 对单条未发货订单给出“满足发货条件”提示：
  - 在库存管理页提示
  - 在发货登记页提示
- 当前阶段仍不做复杂库存流水、锁库、安全库存与采购建议。

### 涉及模块
- 数据库层：
  - 库存表
  - 可选库存变更记录表
- UI 层：
  - 库存维护页
  - 发货页提示区
  - 查询页辅助状态
- 业务逻辑层：
  - 未发订单物料需求汇总
  - 单订单可发货判定
  - 页面级提示

### 数据结构依赖
- 强依赖 Milestone 5A 的订单组件快照。
- 库存按物料维度统计，不按型号维度统计。
- 可发货判定基于“订单当前未发组件需求”与“现有库存数量”的逐项比较。

### UI 任务拆分
- 库存管理页：
  - 物料库存录入
  - 当前库存列表
  - 未发需求页：
    - 未满足发货条件订单列表
    - 单订单 / 全部订单缺料查看
  - 可发货订单提示列表
- 发货登记页：
  - 选中订单时显示是否满足整单发货条件
  - 支持整套发货与单组件发货
- 查询页：
  - 增加“可发货 / 不可发货”辅助状态
  - 显示结构化发货历史
  - 拆分为“订单查询 / 订单详情”两个子页
  - 点击订单后在详情页查看最终订单组件与发货记录
  - 详情页中组件表与发货记录表按上下结构展示

### 开发顺序
1. 建库存主表。
2. 做库存录入和库存列表。
3. 做未发订单物料需求计算。
4. 做单订单可发货判定。
5. 接入库存页与发货页提示。
6. 接通结构化发货、库存扣减与发货历史。
7. 最后再评估库存流水、锁库、扣减策略。

### MVP（最小可运行路径）
- 手工维护当前库存数量。
- 系统可基于未发订单组件快照计算未发需求。
- 系统可对单条订单显示：
  - 可发货
  - 不可发货
- 库存管理页和发货登记页都能看到该提示。
- 当前已支持：
  - 发货时自动扣减库存
  - 订单级和组件级发货记录
- 当前不要求：
  - 自动锁定库存
  - 安全库存预警
  - 采购建议
  - 仓位管理

### 完成标准
- 库存按物料维度维护。
- 未发订单物料需求统计正确。
- 对任一未完成订单，系统可判断是否满足整单发货条件。
- 库存页与发货页均可显示提示。
- 支持结构化订单整套发货与单组件发货。
- 发货成功后库存数量同步扣减。
- 查询页可稳定显示订单基础信息，不再因排序时机出现日期、客户、基础配置等列偶发空白。

# OrderManagerSystem 分阶段实施计划

## 规划原则
- 遵循 `requirements.md` 与 `AGENTS.md`，使用 C++、Qt 6、Qt Widgets、CMake、SQLite。
- 按“先跑通最小闭环，再补充功能”的顺序推进，避免一次性铺开全部能力。
- 当前阶段只规划最小可运行版本，不提前引入导出、权限、网络同步、复杂报表等后续能力。

## Milestone 1：数据库初始化与建表

### 当前状态
- 已完成并通过当前范围内验证。

### 目标
- 程序启动时能够自动创建 SQLite 数据库并完成核心表初始化。
- 明确订单、订单组件、模板、模板组件、发货记录之间的关系字段。
- 保证重复启动不会重复建表报错，为后续录单和发货提供稳定数据基础。

### 范围
- 初始化数据库连接。
- 建立最小核心表：
  - `product_models`
  - `option_templates`
  - `option_template_components`
  - `order_items`
  - `order_item_components`
  - `shipment_records`
- 约定主键、外键关联字段和必要的数量字段。
- 视需要写入最小演示数据，便于后续界面联调。

### 涉及文件
- `databasemanager.h`
- `databasemanager.cpp`
- `main.cpp`
- `CMakeLists.txt`

### 完成标准
- 应用启动后可自动创建数据库文件。
- 核心表结构集中在数据库管理层维护，不分散在 UI 代码中。
- 数据表设计能够支撑模板展开为订单快照，且保留发货历史。

### 验证方式
- 构建验证：
  - `cmake -S . -B build`
  - `cmake --build build`
- 运行程序两次，确认数据库初始化成功且重复执行不报错。
- 使用 SQLite 查看表结构，确认核心表已创建且关键字段齐全。

## Milestone 2：订单录入最小闭环

### 当前状态
- 已完成并通过当前范围内验证。

### 目标
- 打通“录入订单 -> 选择模板或自定义配置 -> 展开订单组件 -> 保存订单”的最小闭环。
- 明确订单项是最小业务单位，模板仅作为录单辅助，最终以订单组件快照落库。
- 让用户可以完成最基本的订单录入与保存，不依赖发货和复杂查询功能。

### 范围
- 在主界面提供最小可用的订单录入区域。
- 支持选择产品型号与配置模板。
- 支持自定义配置模式下按当前产品型号选择可选组件，并填写数量。
- 保存订单主记录到 `order_items`。
- 保存订单实际组件到 `order_item_components`。
- 处理同名组件合并、总需求数量计算等最小业务规则。

### 涉及文件
- `mainwindow.h`
- `mainwindow.cpp`
- `mainwindow.ui`
- `databasemanager.h`
- `databasemanager.cpp`
- `main.cpp`

### 完成标准
- 能录入一条订单并成功保存。
- 使用模板配置时，模板组件能展开为订单实际组件。
- 使用自定义配置时，组件数据只写入订单，不回写模板。
- 历史订单保存后不受后续模板变更影响。

### 验证方式
- 构建验证：
  - `cmake -S . -B build`
  - `cmake --build build`
- 通过界面分别录入：
  - 一条模板配置订单
  - 一条自定义配置订单
- 检查数据库，确认：
  - `order_items` 有对应订单记录
  - `order_item_components` 有正确的组件快照与数量

## Milestone 3：发货功能

### 当前状态
- 已完成并通过当前范围内验证。
- 当前实现已补充：
  - 订单级与组件级发货追加保存
  - 机体独立追踪
  - 订单完成状态判断
  - 机体价格、组件价格、单套总价快照
  - 录单区“主体单价 / 单套总价”拆分显示

### 目标
- 实现订单级和组件级发货登记，支持多次追加发货并保留历史。
- 能实时看到已发数量与未发数量，满足基础跟踪需求。
- 保持“追加记录”模式，不用覆盖历史数据。

### 范围
- 增加发货录入界面或页面。
- 支持按订单登记发货数量。
- 支持按订单组件登记发货数量。
- 将每次发货保存为独立记录。
- 更新订单和订单组件的已发/未发数量字段或可计算状态。

### 涉及文件
- `mainwindow.h`
- `mainwindow.cpp`
- `mainwindow.ui`
- `databasemanager.h`
- `databasemanager.cpp`

### 完成标准
- 对同一订单可多次追加发货。
- 发货后历史记录可追溯，不被覆盖。
- 订单级和组件级未发数量能正确变化。
- 不因一次发货破坏原始订单组件快照。

### 验证方式
- 构建验证：
  - `cmake -S . -B build`
  - `cmake --build build`
- 对同一订单执行多次发货登记，检查：
  - `shipment_records` 中有多条历史记录
  - `order_items` 与 `order_item_components` 的已发/未发数量正确
- 验证超发、空发货等异常输入被界面或逻辑层拦截。

### 目标

### 涉及文件

### 完成标准

- 查询结果能联动查看订单组件与发货情况。
- 页面仍保持轻量、清晰的 Qt Widgets 风格，并与录单、发货页签分离。

### 验证方式

## Milestone 5：基础资料维护

### 当前状态
- 未实现。
- 当前产品型号、产品组件库、模板、模板组件主要依赖初始化数据与演示数据，不具备正式业务使用所需的维护入口。

### 目标
- 在保持现有固定业务表结构不变的前提下，补齐可正式投入使用的基础资料维护能力。
- 让系统能够在不依赖预置演示数据的情况下，维护产品型号、产品组件库、模板和模板组件。
- 为后续导入能力、新数据库启用和正式订单跟踪提供稳定的主数据入口。

### 范围
- 产品型号维护：
  - 新增产品型号
  - 编辑产品型号
  - 维护主体默认价格
  - 启用 / 停用
  - 删除前引用校验
- 产品组件库维护：
  - 为指定产品新增组件
  - 编辑组件名称
  - 编辑默认单价
  - 启用 / 停用
  - 删除前引用校验
  - 同一产品下组件名称唯一校验
- 模板维护：
  - 为指定产品新增模板
  - 编辑模板名称
  - 启用 / 停用
  - 删除前引用校验
  - 模板复制
- 模板组件维护：
  - 为模板新增组件行
  - 从当前产品组件库选择组件
  - 设置每套数量
  - 设置组件单价
  - 删除组件行
  - 同一模板内组件名称重复校验

### 涉及文件
- `mainwindow.h`
- `mainwindow.cpp`
- `mainwindow.ui`
- `databasemanager.h`
- `databasemanager.cpp`
- 数据模型相关头文件 / 结构定义

### 完成标准
- 不依赖演示数据，也能通过界面维护完整基础资料。
- 停用的数据默认不再进入录单下拉，但历史订单显示不受影响。
- 删除操作均具备引用校验，不允许破坏历史订单、模板或发货记录。
- 同一产品下组件名称唯一、同一产品下模板名称唯一、同一模板内组件名称唯一。
- 模板复制后可形成新的独立模板，不影响原模板与历史订单快照。

### 验证方式
- 构建验证：
  - `cmake -S . -B build`
  - `cmake --build build`
- 通过界面分别验证：
  - 新增 / 编辑 / 停用产品型号
  - 新增 / 编辑 / 停用产品组件
  - 新增 / 编辑 / 复制 / 停用模板
  - 新增 / 编辑 / 删除模板组件行
- 验证引用校验、唯一性校验和停用后录单下拉过滤行为符合预期。

## Milestone 6：新数据库初始化与基础资料导入

### 当前状态
- 未实现。
- 当前系统可在现有数据库上继续运行，但尚未形成“从空数据库建立业务表并导入基础资料后投入使用”的正式能力。

### 目标
- 支持新建一个新的可用业务数据库，并自动初始化当前系统固定业务表结构。
- 在此基础上支持导入产品型号、产品组件库、模板及模板组件基础资料，再开展录单、发货、查询和后续汇总业务。
- 明确该能力是面向当前系统自有固定表结构的初始化与导入，而不是适配任意第三方数据库结构。

### 范围
- 新数据库初始化：
  - 新建数据库文件
  - 自动创建当前固定业务表
  - 写入必要系统级基础数据或默认值
- 产品型号导入：
  - 支持 CSV 导入
  - 最小字段包括：产品名称、主体默认价格、启用状态
  - 校验产品名称非空、默认价格合法、同名重复处理
- 产品组件库导入：
  - 支持 CSV 导入
  - 最小字段包括：产品名称、组件名称、默认单价、启用状态
  - 校验产品必须存在、同一产品下组件名不可重复、价格必须合法
- 模板与模板组件导入：
  - 支持一种最小可行的模板导入方案
  - 实现前需在“单个平铺 CSV 导入”和“模板头 / 模板组件双文件导入”之间做选择，并明确理由
  - 模板导入需至少覆盖以下必要信息：
    - 产品名称
    - 模板名称
    - 组件名称
    - 每套数量
    - 组件单价
  - 如方案包含模板头信息，还应支持：
    - 启用状态
    - 排序值 / 常用标记
  - 校验同一产品下模板名唯一、同一模板内组件名称唯一、组件必须能在当前产品组件库中定位
  - 模板导入或后续覆盖更新不得影响历史订单组件快照
- 导入预检查与结果反馈：
  - 导入前校验缺失字段、重复数据、非法价格、非法数量、关联不存在
  - 导入结果区分成功、跳过、失败并输出原因
- 新数据库初始化后的正式业务使用流程不应依赖预置演示数据，演示数据仅用于开发联调或演示场景

### 涉及文件
- `mainwindow.h`
- `mainwindow.cpp`
- `mainwindow.ui`
- `databasemanager.h`
- `databasemanager.cpp`
- 与 CSV 解析、导入结果模型相关的文件

### 完成标准
- 用户可从空数据库开始完成“建库 -> 建表 -> 导入基础资料 -> 开始业务使用”的最小闭环。
- 导入仅面向当前系统固定业务表结构，不引入外部数据库结构适配。
- 模板导入或后续覆盖更新不会影响历史订单，因为历史订单仍基于 `order_item_components` 快照保存。
- 导入失败时能够明确返回到行级或记录级错误原因。

### 验证方式
- 构建验证：
  - `cmake -S . -B build`
  - `cmake --build build`
- 使用空数据库执行：
  - 初始化业务表
  - 导入产品
  - 导入产品组件库
  - 导入模板与模板组件
- 导入完成后执行录单、发货、查询，确认基础资料可直接投入使用。
- 验证重复数据、非法价格、非法数量、缺少关联时的预检查与错误反馈。

### 当前状态

### 范围

- 支持常用筛选条件：
  - 客户

### 涉及文件
- `mainwindow.h`
- `mainwindow.cpp`
- `mainwindow.ui`
- `databasemanager.h`
- `databasemanager.cpp`

- 汇总结果基于筛选后的订单集合计算，不修改订单、订单组件和发货记录。

### 验证方式

  - `cmake -S . -B build`
